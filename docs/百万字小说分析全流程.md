# 百万字小说进入分析系统 — 完整流程

> 目标读者：AI 系统（用于理解全链路）
> 更新日期：2026-06-10
> 前提：用户已完成 API 配置（apiKey / model / baseURL 存入浏览器 localStorage）

---

## 流程总览

一本 100 万字的网文进入系统后，经历 **三阶段、约 20-40 次 AI 调用**，最终输出一份六层 DNA 和一个技术样本库。

```
上传 (.txt)
  │
  ▼
═══════════ 阶段 A：基础分析管线（8 步串行） ═══════════
  Step 0  智能切片        → SemanticSlice[] (~50-200 个切片)
  Step 1  文风提取        → styleProfile (markdown, ~3000-8000 字)
  Step 2  叙事动力学/情节 → plotReport (markdown, ~3000-6000 字)
  Step 3  角色动力学      → characterDynamics (markdown, ~2000-4000 字)
  Step 4  读者体验   ╮
  Step 5  叙事约束   ╯  并行 → readerExperience + narrativeConstraints
  Step 6  样本选取        → RepresentativeSample[] (3-5 个)
  Step 7  DNA 压缩(V1)    → novelDna (YAML, 旧版扁平结构)
  │
  ▼
═══════════ 阶段 B：道/气深度分析（DAG 并行） ═══════════
  Phase 1  体验流标注     → 多 reader persona 并行投票
  Phase 2  消融测试 ╮
           势能追踪 ╯ 并行 → 四分类 + 势能周期 + 节奏画像
  Phase 3  DNA V2 构建    → 六层 DNA (程序化，无需 AI)
  Phase 4  技术样本库     → 8 类 × 3 样本 (程序化)
  │
  ▼
═══════════ 阶段 C：写作生成（5 层层次化） ═══════════
  Layer 1  全书大纲       → ~200 字
  Layer 2  阶段规划       → 4-6 个 Phase × ~500 字
  Layer 3  分卷           → N 个 Volume × ~300 字
  Layer 4  章节集合       → M 个 ChapterSet × ~200 字
  Layer 5  每章计划       → K 个 ChapterPlan × ~100 字
  │
  ▼
  章节正文生成 → 6 维审查 → 修正循环（最多 3 轮）
```

## 阶段 A：基础分析管线

### Step 0 — 智能切片

**输入**：`raw.txt`（100 万字清洗后文本）

**过程**：
1. 系统检查文本长度。100 万字远超 6500 字阈值，触发 AI 切片。
2. 将全文发送至 `/api/source/process/slice`，maxTokens=8192。
3. AI 将小说按语义边界切分为片段，每个片段标注：
   - 语义标签（如 `["战斗", "修炼", "转折"]`）
   - 情节弧线归属
   - 情绪基调
   - 叙事功能（setup / escalation / climax / cooldown / resolution）
   - 紧张度（0-10）
4. 若 AI 输出无法解析，使用 `fallbackSlice()` 按固定长度硬切分。

**产出**：`slices.json` — 约 50-200 个 `SemanticSlice` 对象。
**持久化**：每切片完成后立即 `POST /api/library/save-step` 写入磁盘。

**100 万字场景**：文本太长，单次 AI 调用无法处理。但实际上 Step 0 的 prompt 设计是将全文传入并要求 AI 输出"切片标记"，而非让 AI 阅读全文。如果模型上下文窗口不足以容纳 100 万字，系统当前的行为是将全文传入（可能截断）。这是一个已知限制——未来需要先做程序化预切分再 AI 精修。

---

### Step 1 — 文风提取

**输入**：`SemanticSlice[]`（所有切片内容拼接为 markdown）

**过程**：
1. `computeStyleBatches()` 计算是否需要分批。
   - 公式：`maxChars = max(10000, floor(maxContextTokens * 1.5) - 8000)`
   - 100 万字情况下几乎必定分批（可能 5-20 批）
2. 第一批：发送所有切片内容（或第一批内容）→ AI 生成初始文风档案。
3. 后续批次：发送"上一批报告 + 新切片块"→ AI 增量精炼。
4. 每批调用 `/api/source/process/style`，maxTokens=8192。

**产出**：`style_profile.md` — 9 维度文风分析：
- 基础六维：句式、修辞、词汇、叙事视角、对话风格、描写手法
- 增强三维：章节首尾技术、叙述/场景比例、紧张语言标记
- 每个规律附带：支持证据 + 反例 + 适用率

---

### Step 2 — 叙事动力学（情节提取）

**输入**：`SemanticSlice[]` + `styleProfile`

**过程**：
1. 依赖 Step 1 的产出（`styleProfile` 作为上下文传入 prompt）。
2. 同样支持多批次。
3. 调用 `/api/source/process/plot`。

**产出**：`plot_report.md` — 情节结构分析：
- 大情节框架（主线节点映射）
- 小情节模式库（重复出现的叙事模式）
- 伏笔手法
- 节奏规律

---

### Step 3 — 角色动力学

**输入**：`SemanticSlice[]` + `plotReport`

**过程**：依赖 Step 2 的叙事动力学报告。多批次支持。

**产出**：`character_dynamics.md` — 角色关系拓扑 + 角色功能分类。

---

### Step 4 + Step 5 — 读者体验 ‖ 叙事约束（并行）

**这是本次重构的优化点之一**。原为串行，现改为 `Promise.all` 并行执行。

**Step 4 输入**：`SemanticSlice[]` + `styleProfile` + `plotReport` + `characterDynamics`

**产出**：`reader_experience.md` — 读者心理分析：
- 心理缺失模型（小说满足了什么心理需求）
- 高潮势能逆向追踪（势能类型：Cognitive / Emotional / Mixed）
- 综合翻页驱动力

**Step 5 输入**：`SemanticSlice[]` + `plotReport` + `characterDynamics`

**产出**：`narrative_constraints.md` — 叙事约束：
- 冷却机制
- 风险模式
- 禁忌事项

---

### Step 6 — 样本选取

**输入**：切片摘要（仅标题+标签+情绪，不含全文）+ 前序报告

**产出**：`samples.json` — 3-5 个代表性切片（旧版格式）。

> **注意**：此步骤将在阶段 B 被 `技术样本库` 取代（8 类 × 3 样本 = 24 个样本）。

---

### Step 7 — DNA 压缩 (V1)

**输入**：全部 5 份分析报告（不含切片内容）

**产出**：`novel_dna.yaml` — 旧版扁平结构 DNA（保留兼容）。

> **注意**：新版六层 DNA 在阶段 B 由 `dao-pipeline.ts` 程序化构建，不经过 AI。

## 阶段 B：道/气深度分析（新增）

> **触发**：用户在 UI 点击「深度分析」按钮（调用 `useSourceProcessingStore.startDaoAnalysis()`）。
> **编排**：`src/lib/source-processing/dao-pipeline.ts` — DAG 并行执行。

### Phase B1 — 体验流标注

**目标**：不是分析"作者写了什么"，而是记录"读者阅读时的主观体验"。

**过程**：
1. 系统内置 3 个读者 persona：
   - `casual`（随便看看的普通读者）
   - `immersive`（追求沉浸感的读者）
   - `fast_paced`（追求快节奏爽感的读者）
2. **3 个 persona 并行**发起 AI 调用（`Promise.allSettled`）。
3. 每个 persona 对每个切片输出四维分数：
   - `immersion`（沉浸感 1-10）
   - `emotionalIntensity`（情绪强度 1-10）
   - `anticipation`（期待感 1-10）
   - `perceivedPace`（节奏感知 fast/medium/slow）
4. `aggregateExperienceCurves()` 聚合多 persona 结果：
   - 计算每个切片的平均分数
   - 计算标准差（衡量主观性——标准差大 = 读者分歧大）
   - 标记"体验高点"（多 persona 共识 immersion ≥ 7 且标准差 ≤ 2）

**产出**：
- `ExperienceAnnotation[]`（原始标注，每个 persona 一份）
- `ExperienceCurve[]`（聚合后的体验曲线）

**API**：`POST /api/source/process/experience-annotation`（对每批切片调用）

**100 万字场景**：50-200 个切片，3 个 persona × 可能的 2-3 批 = 6-9 次 AI 调用。每次约 5-10 秒。

---

### Phase B2 — 消融测试 + 势能追踪（并行）

#### 消融测试

**目标**：对每个切片执行"消融测试"——删掉这段，会失去什么？

**四分类**：
| 分类 | 含义 | 删除后果 |
|------|------|---------|
| `bone`（骨架） | 必须存在 | 剧情崩 |
| `muscle`（肌肉） | 增强体验 | 体验下降但剧情不崩 |
| `filler_a`（体验填充） | 维持"气" | 阅读节奏被打乱 |
| `filler_b`（机械填充） | 纯凑字数 | 什么都不变 |
| `uncertain`（不确定） | **默认分类** | AI 拿不准时归类于此 |

**关键设计**：
- 默认输出 `uncertain`，宁可承认不确定也不强行归类
- 遵循奥卡姆剃刀：无明确证据则不推断深意
- 过渡段落的默认解释是"管理节奏"

**API**：`POST /api/source/process/ablation-testing`

#### 势能追踪

**目标**：追踪每个高潮的"积累→释放"周期。

**核心公式**：`爽感 = 势能积累 × 释放效率`

**追踪内容**：
- 高潮类型（情感/认知/权力/身份/关系）
- 积累起点（sliceIndex）
- 积累时长（跨越多少切片）
- 中途强化点（作者在等待期间怎样加强期待）
- 回报倍率（low / medium / high / extreme）

**节奏画像**（全局统计）：
- 推进场景比例
- 蓄力场景比例
- 释放场景比例
- 呼吸场景比例
- 存在场景比例
- 校准场景比例

**API**：`POST /api/source/process/tension-tracking`

**并行**：消融测试和势能追踪通过 `Promise.all` 并行执行。

### Phase B3 — DNA V2 构建（程序化，无需 AI）

**输入**：前序所有产出（切片 + 5 份报告 + 体验曲线 + 消融结果 + 势能分析）

**过程**：`buildNovelDNA()` 函数程序化构建六层结构：

```
┌─────────────────────────────────────────┐
│ 道 (dao)                                │
│ ├─ 主情绪场（从体验曲线高点推断）         │
│ ├─ 备选解读                             │
│ ├─ 读者为何停留                         │
│ ├─ AI 置信度                            │
│ └─ 用户确认标记                         │
├─────────────────────────────────────────┤
│ 气 (qi)                                 │
│ ├─ 维持方法列表（从消融+体验推断）        │
│ ├─ 呼吸周期描述（从势能追踪）            │
│ ├─ 节奏画像                             │
│ └─ 破坏因素                             │
├─────────────────────────────────────────┤
│ 结构 (structure)                        │
│ ├─ 骨（bone 消融结果）                  │
│ ├─ 肉（muscle 消融结果）                │
│ ├─ 体验填充（filler_a）                 │
│ └─ 机械填充（filler_b）                 │
├─────────────────────────────────────────┤
│ 引擎 (engines)                          │
│ ├─ 势能模式列表                         │
│ ├─ 角色决策模型                         │
│ └─ 信息控制方式                         │
├─────────────────────────────────────────┤
│ 风险 (failure_modes)                    │
│ ├─ 机械填充过多 → 直接模仿会注水         │
│ ├─ 势能积累周期过长 → 不适合快节奏平台    │
│ └─ 主情绪场为 AI 推断，未经人工确认       │
├─────────────────────────────────────────┤
│ 风格引擎 (style_engine)                 │
│ └─ 引用原始文风报告                      │
└─────────────────────────────────────────┘
```

**主情绪场推断逻辑**（`inferEmotionalField()`）：
1. 找到体验高点切片
2. 统计这些切片的情绪基调
3. 结合平均期待感分数推断
4. 置信度 = 体验高点数量 > 3 则 0.7，否则 0.4

**"道"的自动发现**：系统不分析"作者想表达什么"，而是从"读者在哪些段落体验最好"反推主情绪场。当"道"是"陪伴、打发时间"时，体验高点的特征是：中等沉浸感 + 低紧张度 + 慢节奏。

---

### Phase B4 — 技术样本库（程序化）

**目标**：替代旧版 3-5 个代表性样本，按叙事技术分类采集。

**分类采集**（每类选 3 个）：

| 类别 | 筛选条件 |
|------|---------|
| **hooks**（开头钩子） | `narrativeFunction === 'setup'` 或 `index ≤ 3` |
| **climaxes**（高潮） | `tensionLevel ≥ 8` 或 `narrativeFunction === 'climax'` |
| **buildups**（蓄力） | `tensionLevel 5-8` 且 `narrativeFunction === 'escalation'` |
| **transitions**（转场） | `narrativeFunction === 'cooldown'` 或 `setup` |
| **relationship**（关系推进） | 标签含 `['关系', '情感', '互动']` |
| **worldRevelations**（世界观揭露） | 标签含 `['世界观', '设定', '揭示']` |
| **breathPassages**（呼吸段落） | 消融结果 `filler_a` 或低紧张度非骨架段落 |
| **foreshadowings**（伏笔） | 标签含 `['伏笔', '铺垫', '暗示']` 或有依赖关系 |

## 阶段 C：写作生成

### 上下文组装 (v2)

`assembleLayerContext()` 根据目标层级组装不同上下文：

| 层级 | plotGuide 取什么 | 道/气上下文 | 节奏处方 |
|------|-----------------|------------|---------|
| L1 大纲 | 大情节框架 | 精简版（300 字） | 精简版 |
| L2 阶段 | 节奏+伏笔规则 | 完整版 | 精简版 |
| L3 分卷 | 小情节模式库 | 完整版 | — |
| L4 章节集合 | 小情节模式库 | 完整版 | — |
| L5 每章计划 | 小情节模式库 | 完整版 | **完整版** |

### 5 层生成（先框架后填充）

每层先生成框架（如"有哪些阶段"），再逐个填充细节。道/气上下文和节奏处方已注入所有层的 prompt。

### 章节正文生成

**生成 prompt 注入内容**：
- 文风档案（完整）
- 道/气上下文（主情绪场 + 维持方法）
- 节奏处方（势能模式 + 骨肉填充比例 + 避坑指南）
- 层级上下文（大纲→阶段→卷→集合→计划）
- 前情提要（**已修复**：前一章末尾 1200 字 + 状态增量摘要）

### 6 维审查（已扩展）

| 维度 | 检查内容 |
|------|---------|
| `style_consistency` | 文风是否一致 |
| `plot_coherence` | 情节逻辑是否通顺 |
| `pacing` | 节奏是否符合指令 |
| `pattern_execution` | 小情节模式是否执行 |
| `foreshadow_execution` | 伏笔是否落实 |
| **`qi_consistency`**（新增） | **"气"是否维持——读者是否愿意继续待在这个世界里** |
