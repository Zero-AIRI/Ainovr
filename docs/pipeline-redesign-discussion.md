# 分析管线重构讨论记录

## 概述

本文档记录了 Ainovr 分析管线重构的完整讨论过程——从三份 AI 审查报告的交叉分析，到两轮 grill-me 深度追问，最终收敛为"双层切片 + 全书记忆图谱"的统一管线架构。

---

## 第一阶段：三份 AI 审查报告的交叉分析

### 报告来源

- **报告 1（英文）**：架构洞察最深，指出客户端编排是最大瓶颈
- **报告 2（中文）**：测试用例和代码示例最扎实，直接可落地的样板代码
- **报告 3（中文）**：与前两者高度重叠

### 三者共识（7 项必做）

1. 引入 Zod 做运行时校验，替代 regex 解析 LLM 输出
2. 提取魔法数字为常量文件
3. 修复空 catch 块，至少加日志
4. 补充单元测试，优先覆盖 `text-cleaner`、`safe-path`、`program-indexer`
5. 拆分大文件（`LayerGenerationView`、`SourceNovelDetailView`、pipeline 文件）
6. Repository 模式抽象文件系统访问
7. API route 统一错误处理，结构化错误响应

### 取舍原则

- **以报告 1 定方向，报告 2 填细节**
- 报告 1 的"客户端编排迁移到后端"暂缓——本地部署场景下客户端编排可接受
- 报告 3 的 monorepo/OpenTelemetry 暂不采纳——当前单仓规模不需要

### 用户特别约束

- **不设硬编码字数阈值**（原方案建议 200 万字分流）
- **仅考虑本地部署**（不需要后端 job 队列、对象存储、多用户支持）

## 第三阶段：Grill-me 深度追问

### 第一轮：架构方向（12 项决策）

| # | 问题 | 用户回答 |
|---|------|---------|
| 1 | 三条管线的关系 | **合并为一条，一键到底** |
| 2 | 事件提取 vs 切片顺序 | **事件提取在前**，切片 AI 可参考事件信息 |
| 3 | 实体识别方式 | 事件提取 AI 自行识别 |
| 4 | 道/气如何嵌入 | 拆分嵌入各步骤之间 |
| 5 | 体验标注位置 | 放在文风之前，情绪信息喂给后续 |
| 6 | DNA 产出 | 合并为一步，单一输出 |
| 7 | 样本选取 | AI 选，但能看到事件数据 |
| 8 | 消融 vs 势能 | 并行 |
| 9 | 技法样本库 | 合并到 DNA 步骤 |
| 10 | DNA 构建方式 | 纯代码填充 + AI 定性归纳 |
| 11 | 事件提取窗口切分 | 按字符数直接截窗口，不依赖章节 |
| 12 | 用户配置 | 零配置，一键到底 |

### 用户关键追问

**Q：按叙事节拍切分——如何界定？为什么是 5000 字？**

A（代码核实）：Prompt 告诉 AI "以情节完整性为最高优先级，不在高潮中间切断，以章节边界为自然切割点"。5000 是默认值，允许 ±30% 浮动。2000~30000 字均可配置。

**Q：自动分批为什么不在切片步骤？**

A：切片步骤的输入是全文，输出是多个切片。分批出现在 Step 1-5，因为此时需要把所有切片拼接发 AI，拼接后可能超上下文。

**Q：候选实体如何确认界定？能包括所有吗？**

A：已有 `entity-classifier.ts` 用 AI 分类候选实体（character/location/organization/artifact/concept），只是未接入管线自动流程。最终决定由事件提取 AI 自行识别。

### 用户推动架构转向

> "切片是为了解决模型一次性无法输入长文的问题。你的问题（200 万字时 AI 读不下全文，切片怎么办）如何理解？"

→ 用户指出我搞反了因果关系。切片本身就是解决长文的方案。真正的问题是：第一遍切片如何在不读全文的情况下做？

> "第一次切片可以是小切片，第二次是大段分析的大切片，带着第一次切片分析后的全书记忆图谱，分析大段原文切片，进入后续分析流程。"

→ 用户提出**双层切片架构**

## 第四阶段：双层切片架构细化

### 用户的核心设计

```
第一遍（小切片 ~几章/几千字）：建记忆
  逐切片 AI 提取事件 → 合并为全书记忆图谱

第二遍（大切片 ~按模型上下文）：深度分析
  每个大切片 + 全书记忆图谱 → AI 深度分析 → 汇总 → DNA
```

### 第二轮 Grill-me 追问

**Q：大事件跨窗口、小事件挤一起——怎么让 AI 正确处理事件粒度？**

> "统一粒度，靠因果关系链连接。"

→ 不区分"大事件"和"小事件"，每个事件按统一标准提取。跨切片的大事件通过因果链（causes/effects）连接。

**Q：切片 AI 看不到跨切片因果——causes/effects 怎么填？**

> "切片内因果 + 链接时跨切片。"

→ 小切片 AI 只标注切片内能确认的因果。跨切片因果交给 Step 3 链接 AI 统一建立。

**Q：每个大切片分析产出什么？**

> "每个大切片都分析所有维度，然后再总结。会不会有可能文章作品前后文风不一致的问题。事件很多时候都没有结局，有的是坑、伏笔，后面作者都忘记了。给我你的建议。"

→ 采纳建议：
1. **文风不一致**：不强制统一，如实记录演化轨迹。每个大切片产出局部观察，汇总阶段做演化分析。
2. **未完成事件/伏笔**：显式标注 open/dangling 状态，不假设一定有结局。

**Q：分析目的是什么？**

> "不是分析师视角，而是逆向工程分析。分析的目的和结果要指导实践，也就是文章的生成。不需要一些套话。如果分析不出来，就不写。"

→ 确立**机制型分析**原则：
- ❌ "文风冷峻克制"（描述型套话）
- ✅ "前 80% 段落句长 5-12 字，对话不加心理描写，高潮段落句长缩至 3-5 字"（可复现的操作）

**Q：667 个独立 AI 调用怎么保证一致性？**

> "统一 schema 约束。"

→ 统一 JSON schema 约束输出格式。事件类型用枚举，人物名从原文精确引用。

**Q：消融测试还需要吗？**

> "也不是纯机制。有的文本可能是作者水字数，不要过度分析。作者和文本也会有错误。消融的意思是在事件上分析，有没有推进剧情、或者舒缓节奏、营造氛围的功效。这个是不是水，是一个很主观的想法。"

→ 消融测试改为**事件功能标注**（推进剧情/舒缓节奏/营造氛围/角色塑造/伏笔/过渡），不做"水字数"硬性判定——而是标注功能后由用户判断。

**Q：汇总 AI 怎么处理矛盾和幻觉？**

未解决——用户在最后提出这个担忧后暂停讨论，要求先整理文档。

## 最终架构：统一分析管线（7 步）

```
                     rawText
                        │
        ┌───────────────┴───────────────┐
        │   第一遍：构建全书记忆图谱      │
        │                               │
        │  Step 1: 小切片（程序化）       │
        │  Step 2: 逐片事件提取（AI并行） │
        │  Step 3: 事件链接（一次AI）     │
        │       → 全局 EventGraph        │
        └───────────────┬───────────────┘
                        │
        ┌───────────────┴───────────────┐
        │   第二遍：深度分析              │
        │                               │
        │  Step 4: 大切片（程序化）       │
        │  Step 5: 逐片深度分析（AI并行） │
        │  Step 6: 汇总+事件功能标注      │
        │       → 全局报告               │
        └───────────────┬───────────────┘
                        │
                  Step 7: DNA 压缩
                (代码填充 + AI 定性)
```

### Step 1：小切片（程序化，无 AI）
- 输入：rawText
- 按灵活尺寸切分（5-10 章或几千字/片）
- 纯代码分割

### Step 2：逐切片事件提取（AI，并行）
- 每个小切片 AI 提取结构化事件和实体
- 统一 JSON schema 约束输出格式
- 实体由 AI 自行识别
- 只标注切片内因果，不猜测跨切片关系
- 失败重试 2 次，仍失败则跳过

### Step 3：事件链接（AI，一次调用）
- 输入：所有 per-slice 事件片段
- AI 构建全局事件图谱：因果链、伏笔配对、事件分组、人物归一化
- 只输出**连接关系**（增量），不重新输出事件本体
- 代码机械合并 Step 2 事件 + Step 3 连接 = 全局 EventGraph

### Step 4：大切片（程序化，无 AI）
- 按模型 maxContextTokens 自适应窗口
- 纯代码分割

### Step 5：逐大切片深度分析（AI，并行）
- 输入：大切片原文 + 全书记忆图谱（压缩摘要格式）
- 机制型逆向工程分析，输出结构化 JSON
- 所有维度一步输出：文风机制、叙事机制、角色机制、情绪机制、体验标注（含多 persona 视角）、约束
- **核心原则**：指导生成实践，不写套话，分析不出来就留 null
- 每个机制结论指向"怎么做"而非"是什么"

### Step 6：汇总 + 事件功能标注（AI，一次调用）
- 输入：所有大切片分析 JSON + 全局 EventGraph + 原文抽样
- 合成全局报告（文风演化轨迹、叙事结构、角色弧线、情绪曲线、约束清单）
- 标注每个事件的叙事功能（推进/舒缓/氛围/角色/伏笔/过渡）
- 标记 open/dangling 事件和未回收伏笔
- **待解决**：如何处理多个大切片分析之间的矛盾和幻觉

### Step 7：DNA 压缩（代码填充 + AI 定性）
- 纯代码：从 EventGraph + 汇总数据计算道/气/结构/引擎的量化部分
- AI 定性：提炼风格引擎、技法库、风险标注
- 一份 DNA 包含全部，机制型输出
- 覆盖旧 `novelDna` / `novelDnaV2` / `techniqueSamples`

## 待解决事项

1. **汇总 AI 的矛盾和幻觉处理**：Step 6 汇总 AI 收到多份大切片分析，可能包含互相矛盾的判断和 AI 幻觉。用户提出需要验证机制，讨论在"汇总 = 验证，不是合并"方案处中断。

2. **大切片分析 JSON schema 的具体字段**：方向已确定（机制型、全维度），具体字段定义待定。

3. **事件功能标注的具体分类**：推进剧情/舒缓节奏/营造氛围/角色塑造/伏笔/过渡——待确认是否完整。

4. **pipeline-types.ts 的完整类型定义**：小切片事件、链接输出、大切片分析 JSON、汇总报告的类型定义。
