## 一、项目概览

**墨韵仿写**是一个 AI 辅助小说风格模仿创作工具。核心流程分为两大阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│  阶段 A：素材分析管线（8 步串行 · 客户端编排 · 服务端流式 AI）    │
│  上传小说 → 语义切片 → 文风/情节/人物/体验/约束 → DNA 压缩      │
├─────────────────────────────────────────────────────────────────┤
│  阶段 B：写作生成管线（5 层层次化 · 先框架后细节 · 审查修正循环）  │
│  大纲 → 阶段 → 分卷 → 章节集 → 每章计划 → 正文生成 → 审查 → 修正 │
└─────────────────────────────────────────────────────────────────┘
```

**技术栈关键特征**：
- 单页应用，无 URL 路由（通过 Zustand `activeView` 切换视图）
- 无数据库，纯文件系统存储（`data/` 目录下 JSON/TXT/MD/YAML）
- 自研 SSE 流式客户端（原生 fetch + ReadableStream），不依赖 AI SDK
- AI 配置（apiKey/model/baseURL）存于浏览器 localStorage，每次请求从客户端传至服务端
- 30+ 个 prompt 模板，用户可自定义覆盖任意 prompt
- 服务端不存储 AI 凭证，API Route 均为无状态流式代理

## 三、AI 集成层

### 3.1 核心流式客户端（`src/lib/ai/providers.ts`）

只有一个核心函数 `chatCompletionStream()`：

```typescript
function chatCompletionStream(options, params): ReadableStream<Uint8Array>
```

**流程**：
1. `validateBaseURL()` — SSRF 防护：检查协议白名单，拒绝 localhost/127.0.0.1/内网地址
2. 构造 OpenAI-compatible 请求体 `{ model, messages, max_tokens, stream: true }`
3. 发送 `POST {baseURL}/chat/completions`，Bearer 认证
4. 读取 `response.body` ReadableStream
5. 解析 SSE 格式：`data: {"choices":[{"delta":{"content":"..."}}]}`
6. 提取 `delta.content` 逐块 enqueue 到返回的 ReadableStream

**关键特征**：
- 不使用 `ai` / `@ai-sdk/openai` 包（它们在 dependencies 中但运行时未使用）
- 原生 `fetch` + `ReadableStream` API
- 通过 OpenAI-compatible 协议兼容任何 API（DeepSeek、GLM 等）
- 默认 maxTokens: 4096（处理管线步骤用 8192，实体分类用 2048，DNA 压缩用 4096）

### 3.2 AI 配置流

```
localStorage (useSettingsStore)
    ↓ (每次请求时读取)
客户端组件 / 管线编排器
    ↓ (作为 JSON body 传参)
Next.js API Route Handler
    ↓ (调用 chatCompletionStream)
OpenAI-compatible API
    ↓ (SSE 流式响应逐块返回)
客户端 ReadableStream → 累积展示
```

**凭证不落盘**：服务端不存储 apiKey，每次请求都从客户端传入。

### 3.3 Prompt 系统

30+ 个 prompt 构建器，每个导出：
- `build*Messages()` — 动态构建 `{ systemPrompt, userMessage }`
- `DEFAULT_SYSTEM_PROMPT` — 默认 system prompt 常量

**Prompt 分类**：
- 素材处理（12 个构建器）：切片、实体分类、文风提取+补充、情节提取+补充、角色动力学+补充、读者体验+补充、叙事约束+补充、DNA 压缩、样本选择
- 层级生成（8 个构建器）：大纲、阶段框架+详细、分卷框架+详细、章节集合框架+详细、每章计划框架+详细
- 章节生成（3 个构建器）：章节写作、审查、修正

**自定义覆盖**：用户通过 PromptManagementView 修改任意 prompt → POST `/api/prompts/save` → 写入 `data/prompts.json`。`usePromptStore.getPrompt(key)` 优先返回自定义版本。

## 四、素材分析管线（8 步串行）

> **编排位置**：`src/lib/store/source-processing.ts` 的 `startProcessing()` 方法
> **状态机**：`raw → slicing → extracting → character_dynamics → deep_analyzing → selecting → compressing → ready`（任何步骤失败进入 `error`）

### 执行流程图

```
用户点击"一键分析"
    │
    ▼
Step 0: 智能切片 (smart-slicer.ts)
    │ POST /api/source/process/slice
    │ Input: raw.txt
    │ Output: SemanticSlice[]
    │ save: slices.json
    │ 状态: 'slicing'
    ▼

    ▼
Step 2: 叙事动力学/情节 (plot-extractor.ts)
    │ POST /api/source/process/plot
    │ Input: SemanticSlice[] + styleProfile
    │ Output: plotReport (markdown)
    │ save: plot_report.md
    │ 状态: 保持在 'extracting'
    │ 依赖 Step 1 的输出
    ▼
Step 3: 角色动力学 (character-dynamics-extractor.ts)
    │ POST /api/source/process/character-dynamics
    │ Input: SemanticSlice[] + plotReport
    │ Output: characterDynamics (markdown)
    │ save: character_dynamics.md
    │ 状态: 'character_dynamics'
    │ 依赖 Step 2 的输出
    ▼
Step 4: 读者体验 (reader-experience-extractor.ts)
    │ POST /api/source/process/reader-experience
    │ Input: SemanticSlice[] + styleProfile + plotReport + characterDynamics
    │ Output: readerExperience (markdown)
    │ save: reader_experience.md
    │ 状态: 'deep_analyzing'
    │ 依赖 Step 1+2+3
    ▼
Step 5: 叙事约束 (narrative-constraints-extractor.ts)
    │ POST /api/source/process/narrative-constraints
    │ Input: SemanticSlice[] + plotReport + characterDynamics
    │ Output: narrativeConstraints (markdown)
    │ save: narrative_constraints.md
    │ 状态: 保持在 'deep_analyzing'
    │ 依赖 Step 2+3
    ▼
Step 6: 样本选择 (sample-selector.ts)
    │ POST /api/source/process/samples
    │ Input: 切片摘要（非全文）+ styleProfile + plotReport
    │ Output: RepresentativeSample[] (3-5 个)
    │ save: samples.json

    │ 仅传摘要以节省 token
    ▼

    │ save: novel_dna.yaml
    │ 状态: 'compressing' → 'ready'
    │ ⚠️ 此步失败不阻塞管线
    ▼

```

### 关键实现细节

**多批次处理**（Steps 1-5 支持）：
- `computeMaxCharsPerBatch(maxContextTokens)` 计算单批最大字符数：`max(10000, floor(maxContextTokens * 1.5) - 8000)`
- `splitIntoBatches()` 按段落/换行/句号边界拆分
- 第一批：`build*Messages()` 生成初始报告
- 后续批次：`build*SupplementMessages(previousReport, newChunk)` 增量精炼

**短文本跳过**：
- Step 0：文本 < 6500 字符时跳过 AI，`createSingleSlice()` 直接创建一个切片

**进度模拟**：
- 每步得 12.5%（100/8）进度条
- 流式期间每 2 秒 +1%
- 标记流完成时瞬间填满该步进度

**增量持久化**：
- 每步完成后立即 `POST /api/library/save-step`，更新 `meta.json` + 写入对应文件
- 刷新页面后可从已有步骤继续（通过 meta.json 的状态标记判断）

**Step -1（未集成到 8 步管线中）**：
- `program-indexer.ts`：纯程序化的中文 n-gram 实体提取（2-4 字）
- `entity-classifier.ts`：AI 分类实体类型（角色/地点/组织/神器/概念）

## 五、写作生成管线（5 层层次化）

> **编排位置**：`src/lib/generation/layer-pipeline.ts` 的函数 + `LayerGenerationView.tsx` 组件
> **核心模式**：每层先"框架"后"填充"（framework → detail）

### 5 层结构

```
Layer 1: 全书大纲     (~200 字)                          ← 单次调用
    │
    ▼
Layer 2: 阶段规划     (4-6 个 Phase, 每份 ~500 字)       ← 框架 + N×详细
    │
    ▼
Layer 3: 分卷         (每 Phase 1-2 卷, 每卷 ~300 字)   ← 框架 + N×详细
    │
    ▼
Layer 4: 章节集合     (每卷 3-5 个 Set, 每个 ~200 字)   ← 框架 + N×详细
    │
    ▼
Layer 5: 每章计划     (~100 字/章)                       ← 框架 + N×详细
    │
    ▼
章节正文生成 → 5 维审查 → 修正循环（最多 3 轮）
```

### 上下文组装器（`context-assembler.ts`）

`assembleLayerContext(layer, sourceNovels, project)` 为每层组装不同上下文：

| 层级 | styleGuide | plotGuide（提取部分） | hierarchyContext | chapterTask |
|------|-----------|----------------------|------------------|-------------|
| L1 | 全部文风源 | 大情节框架（## 一、） | — | — |
| L2 | 全部文风源 | 节奏+伏笔规则（## 三/四、） | 大纲 | — |
| L3 | 全部文风源 | 小情节模式库（## 二、） | 大纲 + 阶段 | — |
| L4 | 全部文风源 | 小情节模式库 | 大纲 + 阶段 + 分卷 | — |
| L5 | 全部文风源 | 小情节模式库 | 大纲 + 阶段 + 分卷 + 集合 | 本章计划 |

**源小说角色**（通过 `SourceRole` 控制）：
- `style`：仅贡献 styleGuide
- `plot`：仅贡献 plotGuide
- `style_and_plot`：两者都贡献

### 每层生成模式（以 Layer 2 阶段为例）

```
1. LayerGenerationView 调用 generatePhaseFramework()
   → POST /api/generation/phase-framework
   → 流式返回阶段框架文本（列出所有阶段名称和章节范围）

2. 解析框架，得到 N 个阶段

3. 对每个阶段调用 generatePhaseDetail(phaseIndex, framework)
   → POST /api/generation/phase-detail
   → 流式返回该阶段的 ~500 字详细描述

4. 保存到 project.phases[]，自动推进 currentLayer
```

Layer 3-5 同理，都是「框架 → N×详细」。

### 每层对应的 API 路由

| 层级 | 框架 | 详细 | Prompt 构建器 |
|------|------|------|-------------|
| L1 大纲 | `/api/generation/outline` | — | `buildOutlineGenerationMessages` |
| L2 阶段 | `/api/generation/phase-framework` | `/api/generation/phase-detail` | `buildPhaseFrameworkMessages` / `buildPhaseDetailMessages` |
| L3 分卷 | `/api/generation/volume-framework` | `/api/generation/volume-detail` | `buildVolumeFrameworkMessages` / `buildVolumeDetailMessages` |
| L4 集合 | `/api/generation/chapter-set-framework` | `/api/generation/chapter-set-detail` | `buildChapterSetFrameworkMessages` / `buildChapterSetDetailMessages` |
| L5 计划 | `/api/generation/chapter-plan-framework` | `/api/generation/chapter-plan-detail` | `buildChapterPlanFrameworkMessages` / `buildChapterPlanDetailMessages` |

### 层次化数据结构

```typescript
Phase { id, index, title, content, volumeCount }
  └→ Volume { id, phaseId, index, title, content, chapterSetCount }
       └→ ChapterSet { id, volumeId, index, title, content, chapterCount }
            └→ ChapterPlan { id, chapterSetId, index, title, content, plotKeywords[], patternStructure }
```

```
├─────────────────────────────────────────────────┤
│ 2. 审查                                          │
│    POST /api/chapter/review                      │
│    5 维度各评分 1-10                              │
│    • style_consistency    文风一致性              │
│    • plot_coherence       情节逻辑               │
│    • pacing               节奏                   │
│    • pattern_execution    小情节模式执行度        │
│    • foreshadow_execution 伏笔/铺垫执行度          │
│    全部 ≥ 7 分 → passed                          │
│    任一 < 7 分 → needs_revision                  │
├─────────────────────────────────────────────────┤
│ 3. 修正（仅 needs_revision 时）                   │
│    POST /api/chapter/revise                      │
│    输入：原文 + 审查问题 + 人工反馈(可选)          │
│    revisionCount++                               │
│    最多 3 轮 → 回到步骤 2                         │
└─────────────────────────────────────────────────┘
```

### 前情提要

最近 5 章内容的前 300 字摘要作为 `previousState` 传入，保持叙事连续性。

1. **Step 4 和 Step 5 可并行**：读者体验和叙事约束的输入依赖不完全重叠，二者都依赖 Step 2+3 但不互相依赖
